Critical Security Fixes

4 RESOLVED
CRIT-01
Oracle Manipulation on MICE Price

Flash loan attack on spot price oracle allows purchasing MICE at 1/10th real cost within a single transaction block.

▶ Decision
Use 24h TWAP minimum from PancakeSwap V3 as primary oracle. Add 10% deviation circuit breaker (freeze if |TWAP_current - TWAP_previous| > 10%). Chainlink as secondary fallback. Never use spot price for any economic calculation.
▶ Rationale
Industry standard. 24h TWAP makes flash loan manipulation economically infeasible (attacker would need to sustain manipulation for 24 hours).
▶ Impacts
MICE License Contract, Hindex Emission Contract, all price-dependent calculations.
CRIT-02
Reentrancy in Reward Distribution

MICE License and NFT Staking contracts vulnerable to reentrancy if state updated after token transfer.

▶ Decision
Enforce Checks-Effects-Interactions pattern on ALL contracts. Add OpenZeppelin ReentrancyGuard modifier on every function that transfers tokens or BNB. No exceptions.
▶ Rationale
Non-negotiable security standard. Every DeFi exploit involving reentrancy could have been prevented by this pattern.
▶ Impacts
All smart contracts with token transfer functionality.
CRIT-03
Admin Single Point of Failure

Single admin private key controls all privileged functions. One compromise = full protocol takeover ($625M Ronin pattern).

▶ Decision
Gnosis Safe 3-of-5 multisig for ALL admin functions. TimelockController: 48h delay on treasury operations, 7-day on parameter changes. OpenZeppelin AccessControl replaces Ownable entirely. Role-based permissions: ADMIN_ROLE, MINTER_ROLE, PAUSER_ROLE, ORACLE_ROLE.
▶ Rationale
Multisig eliminates single-point-of-failure. Timelock gives community visibility before execution. AccessControl provides granular role separation.
▶ Impacts
All smart contracts, deploy scripts, Admin DApp interaction layer.
CRIT-04
L(t) Liquidity Manipulation

Whale temporarily provides massive DEX liquidity to inflate TWAP7d, extracts excess MICE rewards, withdraws liquidity. Risk-free arbitrage.

▶ Decision
Cap L(t) multiplier at max 2.0 (floor 0.5). Enforce minimum liquidity depth to qualify. Whitelist trading pairs only. Add 72h cooldown for new liquidity providers before counted in TWAP.
▶ Rationale
Capped multiplier limits maximum exploitation. Cooldown prevents flash-liquidity attacks. Whitelist prevents synthetic pair manipulation.
▶ Impacts
MICE Emission formula, L(t) calculation module, liquidity monitoring.

Spec Error Resolutions

5 RESOLVED
ERROR-01
Vesting Math Discrepancy (BLOCKING)

White Paper states 6-month cliff + 10% unlock + 2.5%/month. Math: 10% + (36 x 2.5%) = 100% at month 42, not month 38.

▶ Decision
36-month total vesting period confirmed. Schedule: 6-month cliff, 10% TGE unlock at cliff end, 2.5%/month linear over 36 months. 100% vested at month 42 from TGE.
▶ Rationale
Founder decision. 36-month linear provides longer alignment and smoother distribution curve.
ERROR-02
Pre-Sale USDT Allocation Undefined

After F1 (10%) + F2 (5%) referral payouts, remaining 85% USDT had no defined allocation.

▶ Decision
85% split: 45% Treasury (development + operations), 25% Liquidity Provision (DEX), 15% Marketing & Partnerships.
▶ Rationale
Heavy treasury allocation ensures long-term development runway. 25% liquidity provides strong DEX depth at launch.
ERROR-03
Vesting Creation: Manual vs Automatic

Admin DApp implied manual vesting creation. Risk of human error, delays, and manipulation.

▶ Decision
Auto-create atomically at purchase confirmation. SeedSaleContract calls VestingManager.createSchedule() in the same transaction. Admin DApp shows read-only vesting view. Only action: revoke (requires 3-of-5 multisig, fraud/KYC violation only).
▶ Rationale
Eliminates 3 risks: admin forgets to create, admin enters wrong data, manipulation window between purchase and vesting. Atomic = trustless.
ERROR-04
MIC Burn Source Undefined

Admin dashboard showed "MIC Burned" metric but no burn mechanism was defined.

▶ Decision
3 MIC burn sources defined: (1) Marketplace Fee Burn: 1% of all secondary sale value burned permanently. (2) Governance Proposal Burn: 500 MIC deposit burned on failed proposals. (3) Inactivity Reclaim: unclaimed rewards after 180 days returned to protocol treasury. Note: MICE is ERC-1155 NFT — not burnable. Burns apply to MIC token only.
▶ Rationale
Creates sustainable deflationary pressure (~0.8-1.2% annual) without aggressive burning. Governance burn prevents spam proposals. Inactivity reclaim prevents dead allocation.
ERROR-05
DAO Governance Asymmetry

Admin DApp had DAO module but User DApp had no corresponding DAO screen.

▶ Decision
Add DAO placeholder UI in User DApp from Phase 0. Scaffold: Proposal List, Vote Interface, Governance Stats components. Logic activates in Phase 2 but UI structure exists from day one.
▶ Rationale
Early scaffolding prevents breaking changes later. Users see governance is planned from the start, building trust.

17 Clarifying Questions — Answered

ALL RESOLVED
Q01
Mining Supply: Pre-Minted or Dynamic Mint?
▶ Decision
Dynamic mint. MINTER_ROLE on MICToken granted only to authorized emission contracts. Daily E(t) calculated and minted on demand. No pre-mint of mining pool.
▶ Rationale
Keeps circulating supply low initially (good for price). Prevents centralization risk of large pre-minted pool. Transparent: only minted when earned.
Q02
L(t) Formula Complete Definition
▶ Decision
Formula: L(t) = min(2.0, max(0.5, TWAP7d / TWAP_baseline))
Input: TWAP7d from PancakeSwap V3 oracle (7-day time-weighted average MIC/USDT).
Output range: 0.5 to 2.0.
TWAP_baseline: Reference price set by governance (initial = listing price).
Update frequency: Every 24h aligned with daily emission calculation.
Circuit breaker: If |TWAP7d - TWAP_previous| > 10%, freeze L(t) at previous value pending governance review.
Q03
MICE License: ERC-1155 vs Standalone
▶ Decision
ERC-1155. Each tokenId = license tier/batch, amount = number of licenses. Supports batch operations (mint/transfer multiple licenses), native marketplace support, significant gas savings.
▶ Impacts
Unblocks 4 dependent contracts: MICEMining, NFTStaking, Marketplace, Referral reward calculation.
Q03-A
MICE Supply Cap 100,000 & Slot Recycling

Maximum 100,000 MICE licenses may exist simultaneously. New licenses are only issued when existing licenses expire and free up slots.

▶ Decision
Hard cap: MAX_SUPPLY = 100,000. Purchase gate: require(activeLicenseCount < MAX_SUPPLY) checks active licenses, not total minted. When a license expires (lazy evaluation from Q10) → activeLicenseCount-- → slot freed for new buyer.

Public function forceExpireCheck(uint256[] licenseIds) allows anyone to trigger status update on expired licenses, freeing slots. Frontend automates: scan expired → force expire → purchase in one user flow.

Scarcity model: at full capacity (100,000 active), new buyers must either wait for expiry or buy on secondary market at premium. Creates natural renewal pressure — holders who don't renew lose slot to next buyer.
▶ Rationale
Fixed supply + recycling creates real scarcity. Self-balancing: high demand → holders renew → stable ecosystem. Low demand → slots open → easier entry. Only active contributors (high Hindex = high reward) find renewal worthwhile.
▶ Impacts
MICELicense contract (supply cap logic), emission calculation (pool divided by active licenses only), Admin DApp (slot availability dashboard), User DApp (expiry countdown, renewal prompts, slot tracker).
Q04
MFP-NFT Minting Trigger
▶ Decision
Two triggers: (1) Seed Round purchase — admin verifies USDT payment, triggers batch mint per package allocation (20/60/150/350 NFTs). (2) Gold Tier Airdrop — community members meeting Gold criteria eligible for MFP-NFT credential. Approval: Admin verifies on Admin DApp, triggers batch mint, NFTs delivered to buyer wallet.
Q05
MFP-NFT: Soulbound or Transferable?
▶ Decision
Transferable (standard ERC-721). Safeguard: 7-day cooldown after transfer before NFT gains voting power. Prevents DAO accumulation attack (HIGH-06) while preserving investor value.
▶ Rationale
Seed investors need transfer rights as part of investment value. Cooldown mechanism adequately mitigates DAO governance attack vector.
Q06
NFT Staking Reward Formula
▶ Decision
Dynamic, tied to emission pool. Formula:
R_staker(t) = (NFT_multiplier x MIC_staked) / Total_weighted_stake x P_nft x E(t)
P_nft = 20% of daily emission for NFT staking pool. APY variable depending on total stake and emission rate. No fixed APR.
Q07
Vesting Contract: Separate or Embedded?
▶ Decision
Separate VestingManager contract. Reusable across Seed, Pre-Sale, Team vesting, and future rounds. SeedSaleContract calls VestingManager.createSchedule() atomically. Clean separation of concerns.
Q08
Seed Round Payment Methods
▶ Decision
USDT (BEP-20) as primary + BNB accepted with auto-convert via PancakeSwap Router at market rate. USDT is internal unit of account. Contract handles both payment paths.
Q09
KYC Enforcement Model
▶ Decision
Hybrid: Off-chain KYC via Sumsub (speed + UX) + on-chain allowlist mapping for Seed Round (admin whitelists approved wallets). Upgrade to on-chain attestation in Phase 2.
Q10
MICE License Expiry Mechanism
▶ Decision
Lazy evaluation on-chain. No keeper/cron. On any user interaction (claim, transfer, renew), contract checks block.timestamp > expiryDate. If expired, auto-deactivate and remove from emission. 72h grace period for renewal. Zero cost for protocol.
Q11
Hindex Calculation: On-Chain or Off-Chain?
▶ Decision
Off-chain computation, on-chain submission via authorized oracle (multisig protected). Backend calculates Hindex scores from activity data, submits to chain. Merkle proof available for user verification. Phase 3: migrate to Chainlink Functions or ZK proof for trustless verification.
Q12
Referral System Scope
▶ Decision
Pre-Sale ONLY. Seed Round has NO referral program — Seed investors are strategic partners who join for vision, not referral incentives. Pre-Sale referral structure: multi-level F1 (8%), F2 (4%), F3 (2%), F4+ (1%). Referral Registry contract only integrates with PreSaleContract (Phase 1), NOT SeedSaleContract.
▶ Rationale
Simplifies Phase 0: SeedSaleContract has no referral dependency. Reduces attack surface. Referral fits community expansion (Pre-Sale) not strategic fundraising (Seed).
Q13
Referral Commission Payment Model
▶ Decision
Batched off-chain, claimable on-chain via merkle proof. Backend tracks earnings, aggregates weekly, submits merkle root on-chain. Referrers claim when ready. Saves 80%+ gas vs immediate payout.
Q14
Self-Referral Prevention
▶ Decision
Multi-layer defense: (1) KYC linkage — 1 identity = 1 referral tree. (2) IP + device fingerprinting off-chain. (3) Minimum $100 investment for referral bonus activation. (4) 30-day cooldown between same-entity referral registrations. (5) Admin dashboard pattern detection and manual review.
Q15
KYC Provider & Payment Model
▶ Decision
Provider: Sumsub. Best price-to-feature for crypto projects. 220+ countries, BSC/Web3 integration, webhook API, ~$1.5-2/verification. Proven in multiple token sales.

Payment Model C (Hybrid): Seed Round — protocol pays all KYC fees (VIP experience for strategic investors). Pre-Sale ≥ $100 investment — protocol pays KYC fees. Pre-Sale < $100 investment — user pays ~$2 verification fee. Sumsub invoices MissionChain monthly; protocol pays from treasury via fiat.
▶ Rationale
$100 threshold filters spam registrations while keeping barrier low for Global South community. Pre-Sale Standard package = $100, so most serious investors are covered. Seed investors get frictionless VIP onboarding.
Q16
Blockchain Event Indexing
▶ Decision
The Graph subgraph for production (decentralized, BSC-supported, free tier via Subgraph Studio). Custom listener for local development/testing only. No Moralis (vendor lock-in risk).
Q17
TWAP Oracle Source
▶ Decision
PancakeSwap V3 TWAP as primary (built-in observation from pool). Chainlink as secondary/fallback when MIC has sufficient volume for custom feed. Admin-submitted rejected entirely — centralization risk violates protocol principles.
▶ Impacts
Directly resolves CRIT-01. All price-dependent calculations use this oracle stack.
EMISSION ENGINE

Adaptive Emission Engine — Founder Decisions

EM-01
Unified Mining Pool Architecture
▶ Decision
85% of total supply (5.95B MIC) = ONE unified mining pool mined exclusively through MICE licenses. NFTs are staking rights only — no separate emission pools. All emission flows through MICE mining, then split algorithmically.
▶ Impacts
Fundamental architecture of token emission. Simplifies smart contract design. All emission controlled by EmissionController.
EM-02
Emission Split Ratios (60/15/20/5)
▶ Decision
Every MIC emitted is split: 60% MICE Miners (by Hindex), 15% DAO Treasury, 20% NFT Stakers (merged MFP-NFT + Community NFT in one pool), 5% Auto-Burn (to 0xdead). DAO can adjust within guardrails: miners [50-70%], DAO [10-20%], staking [15-25%], burn [3-10%]. Max change ±5%/quarter.
▶ Impacts
Determines reward distribution for all participants. Auto-burn creates permanent deflation (~297.5M MIC over lifecycle).
EM-03
Target ROI: 250% | Halving: 180 Days
▶ Decision
ROI Regulator targets 250% annual ROI for MICE miners. R(t) = clamp(ROI_target / ROI_current, 0.5, 2.0). Halving period T_half = 180 days (aggressive — E_base halves every 6 months). E_0 = 22,907,500 MIC/day. Creates strong early-adopter incentive with rapid transition to utility-driven value after ~18 months.
▶ Impacts
Early miners earn above-target ROI (~290-475%). By month 9, ROI converges to 250%. After month 18, ROI declines — forcing ecosystem to derive value from utility (Mission Learn/Work/Social). DAO can adjust T_half [90d-365d] and ROI_target [100%-500%].
EM-04
Dynamic MICE Pricing ($300–$1,000)
▶ Decision
P_mice(t) = $300 + $700 × U(t), where U(t) = activeMICE/100,000. Floor $300, ceiling $1,000. Revenue split: 50% Protocol Treasury, 30% MIC/USDT Liquidity, 20% MIC Buyback & Burn. Dynamic pricing creates natural supply-demand equilibrium for MICE licenses.
▶ Impacts
Interacts with ROI Regulator: higher MICE price raises ROI denominator, triggering emission increase. Buyback-and-burn from revenue adds ~240M MIC deflationary pressure (at estimated volumes).
EM-05
NFT Staking: Merged Pool, No-NFT at 0.5×
▶ Decision
Single merged staking pool (20% of emission) for all NFT tiers. MFP-NFT: 10× multiplier, 1M MIC cap. Platinum: 5×/500K. Gold: 2.5×/250K. Silver: 1×/100K. Non-NFT holders allowed at 0.5×/50K cap. Time-lock multipliers: 30d=1×, 90d=1.25×, 180d=1.5×, 360d=2×. Staking Ratio Target: 40%. Unstake limit: 10%/day (admin adjustable).
▶ Impacts
Inclusive design (non-NFT can stake) with strong upgrade incentive (0.5× → 1× = 2× effective boost). 10% unstake limit prevents bank runs while remaining accessible. Admin-adjustable for emergency response.
EM-06
5 Circuit Breakers
▶ Decision
CB-1: Cumulative hard cap 5.95B (immutable). CB-2: Daily cap 2× E_base (auto). CB-3: Price floor pause at $0.001 MIC (auto-resume at $0.002/24h). CB-4: Unstake 10%/day (admin adjustable). CB-5: Emergency pause via Gnosis 3-of-5 + 24h timelock for unpause.
▶ Impacts
Layered safety: CB-1/2 automatic, CB-3 automatic with recovery, CB-4 admin-tunable, CB-5 manual emergency only. Ensures system self-regulates in normal conditions.
