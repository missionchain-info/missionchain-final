§ 02

Agent Responsibilities

🦋
CLAUDIA Chief Orchestrator
Orchestrator ▼
  • Receives brief from Thomas L. (Co-Founder)
  • Assigns tasks to teams with clear specs
  • Collects outputs, resolves conflicts between teams
  • Escalates unresolved issues to Thomas
  • Produces final delivery report after each feature
🏗️
ARCHITECT Team Builder
Builder ▼
⚙ Model: Claude Sonnet
  • Read Fullstack Spec + White Paper before any code
  • Design database schema, API structure, contract interfaces
  • Ask clarifying questions before implementation begins
  • Write technical decision log (all design choices documented)
  • Align with Auditor team on design during Kickoff
💻
CODER Team Builder
Builder ▼
⚙ Model: Claude Sonnet
  • Implement code strictly per confirmed spec
  • Smart contracts (Solidity) — MIC Token, MICE, MFP-NFT, Seed Sale
  • Backend API (Node.js/TypeScript + Prisma)
  • Frontend components (React/Next.js)
  • Write inline documentation for every function
  • Respond to Auditor feedback within each review cycle
🔒
SECURITY REVIEWER Team Auditor
Auditor ▼
⚙ Model: Claude Sonnet
  • Review smart contracts per OpenZeppelin security standards
  • Check for: reentrancy, integer overflow, access control, front-running
  • Challenge Architect design decisions — no rubber-stamping
  • Debate directly with Builder if disagreement exists
  • Sign off only when zero CRITICAL + zero HIGH security issues
📐
LOGIC REVIEWER Team Auditor
Auditor ▼
⚙ Model: Claude Sonnet
  • Verify business logic matches White Paper v3.5 exactly
  • Check tokenomics implementation (Hindex, vesting, referral)
  • Validate Admin vs User permission boundaries
  • Identify edge cases in user flows
  • Ensure Admin DApp and User DApp are consistent
🎭
SCENARIO TESTER Team QA
QA ▼
⚙ Model: GPT-4o-mini
  • Simulate 1,000+ user interactions per feature
  • Test all 4 roles: Super Admin / Admin / Normal User / Attacker
  • Generate automated test cases
  • Report bugs in standard format (Severity / Steps / Expected / Actual)
  • Re-test after each Coder fix
⚡
STRESS TESTER Team QA
QA ▼
⚙ Model: GPT-4o-mini
  • Load test: 100 / 500 / 1,000 concurrent users
  • Estimate gas costs for all contract functions
  • Test race conditions (simultaneous SEED purchases)
  • Simulate network failure, timeout, retry scenarios
  • Confirm system degrades gracefully under load
§ 03

Phase 0 — Kickoff & Alignment

⚠️   Rule: Align first. Code never.   ⚠️
1

All Agents Read Source Documents

  • Fullstack Architecture Spec
  • Web3 DApp — Admin + User flows
  • White Paper v3.5 (tokenomics, smart contracts)
  • API specification
2

Each Team Submits Questions Independently

BUILDER "What exactly do I need to build?"
AUDITOR "What could go wrong in this design?"
QA "What could fail under real usage?"
3

Open Debate Session

  • All questions collected; teams respond to each other
  • Claudia moderates the session
  • Thomas has final say on all decisions
4

Decision Log Locked

  • All design decisions documented in writing
  • Every agent signs off: "I understand the spec 100%"
  • Only then does coding begin — no exceptions
§ 04

Development Cycle — Per Feature

Thomas → Claudia  (brief + priority)
[ KICKOFF ]   All teams read spec
Architect designs → Security challenges → Logic validates
Debate: max 2 rounds
✦   Claudia + Thomas confirm design
💻   Coder implements
Security Reviewer + Logic Reviewer audit code
Feedback → Coder fixes → re-review · max 3 rounds
🧪   Scenario Tester + Stress Tester — full test suite
Bugs → Coder fixes → re-test · max 3 rounds
🦋   Claudia produces delivery report
✅   Thomas approves  →  DEPLOY
§ 05

Bug Report Format

Standard format all QA agents must use for every reported issue. 

BUG #[ID]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Severity:    CRITICAL / HIGH / MEDIUM / LOW
Component:   [Smart Contract / API / Frontend / Admin DApp / User DApp]
Role:        Super Admin / Admin / User / Attacker
Feature:     [feature name]

Scenario:    [what was being tested]
Steps:       1. [step]
             2. [step]
             3. [step]
Expected:    [what should happen]
Actual:      [what actually happened]
Impact:      [business / security impact]

Status:      OPEN → FIXED → VERIFIED → CLOSED
Assigned to: CODER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
§ 06

Escalation Rules

Situation Action
Builder + Auditor agree Proceed
Builder + Auditor disagree +1 Debate Round
Unresolved after 3 rounds Claudia decides
Claudia unsure Escalate to Thomas
CRITICAL bug found Stop all work — fix first
CRITICAL bug unresolvable Escalate to Thomas immediately
Max autonomous rounds before Thomas escalation:  3 rounds
§ 07

Definition of Done

A feature is DONE only when all of the following are satisfied:

✓
Zero CRITICAL security issues
✓
Zero HIGH security issues
✓
All business logic matches White Paper v3.5
✓
All 4 test roles pass (Admin / User / Attacker scenarios)
✓
Load test: 1,000 concurrent users without failure
✓
Gas costs estimated and within acceptable range
✓
Code documented — inline + README
✓
Decision log updated with all choices
✓
Thomas approved — final sign-off
